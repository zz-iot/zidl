# zidl — DDS Ecosystem

How zidl fits into a DDS stack, the full-DDS vs XRCE profile distinction,
and how the `--generate-interfaces` option generates the abstract DCPS API layer.

---

## What zidl Is (and Isn't)

zidl is the **IDL compiler and code generator** for a forthcoming Zig-native DDS
stack. It is not itself a DDS implementation. Its artifacts feed into the DDS
runtime:

```
zidl (this repo)
  └── generates: type definitions + CDR serializers + TypeObjects
  └── generates: DDS binding layer (--generate-interfaces)

zidl-rt / zidl-cdr / zidl-xtypes (companion packages in this repo)
  └── runtime CDR encode/decode used by generated code

[future: Zig DDS runtime]
  └── RTPS participant, transport, discovery, QoS
  └── DataWriter / DataReader implementation
  └── TypeObject registration and type discovery
```

The DDS binding layer (`--generate-interfaces`) generates the abstract API
stubs (DataWriter, DataReader, TypeSupport, etc.) that the DDS runtime will
implement as concrete types. The generated stubs define the required interface
shape; the runtime provides the vtable implementations.

## Full DDS vs XRCE

### Full DDS Profile (`--profile full`, the default)

- XCDR2 encoding with `@appendable` and `@mutable` support.
- Unbounded sequences, strings, and maps.
- TypeObject/TypeIdentifier generation (MinimalTypeObject XCDR2 LE streams).
- Heap allocation in generated code.
- Full DCPS API (`--generate-interfaces`): Entity, DomainParticipant, Publisher,
  DataWriter, Subscriber, DataReader, Topic hierarchy.
- Targeting DDS v1.4 (formal/15-04-10) compliance.

### XRCE Profile (`--profile xrce`)

Subset for bare-metal microcontrollers (DDS-XRCE v1.0, formal/20-02-01):

- XCDR1 encoding only (no DHEADER, no EMHEADER).
- Only `@final` types allowed (no `@appendable`, no `@mutable`).
- Only bounded sequences (`sequence<T, N>`); no unbounded sequences.
- No `map<K,V>`.
- No TypeObject/TypeIdentifier.
- No heap allocation in generated code.

XRCE profile constraints are validated before backends run (`validateXrce` in
`src/backend/interface.zig`). Violation is a compile-time error, not a runtime check.

The `--zig-version 0.15.1` flag emits Zig output compatible with MicroZig clients:
bounded strings and sequences use fixed-capacity `zidl_rt.BoundedArray` storage.
A fully freestanding/no-heap runtime path remains planned work — see [`roadmap.md`](roadmap.md).

## Cyclone DDS Interoperability

zidl's CDR output is byte-compatible with Cyclone DDS 11.0.1. The following have
been validated:

- Encap header byte values (`CDR1_LE = 0x0001`, `CDR2_LE = 0x0007`).
- Primitive type encoding (all IDL primitive types).
- String encoding (length-prefixed, NUL-terminated).
- Struct member layout and padding.
- `@appendable` DHEADER framing.
- TypeObject EquivalenceHash byte values.

See [`testing.md`](testing.md) for how to run or regenerate the CDR interop harness.

## TypeObject and Type Discovery

TypeObjects generated by zidl are intended to be registered with a DDS domain
participant and used for type discovery in the DDS-XTypes protocol.

**EquivalenceHash** (`equivalence_hash: [14]u8`): the on-wire identifier used by
Cyclone DDS, FastDDS, and other DDS implementations to match types. zidl-generated
hashes have been validated against Cyclone DDS.

**TypeIdentifier** (`type_identifier: [32]u8`): SHA-256 fingerprint. This is a
zidl-internal convention, not transmitted. Useful for out-of-band tooling (type
registries, schema databases) that needs a stronger identifier than the 14-byte
EquivalenceHash.

**What's deferred**: The `zig_typeobject.zig` encoder handles all five IDL types
(`struct`, `enum`, `union`, `bitmask`, `bitset`), but the Zig backend only emits a
`pub const type_object` field inside generated `struct` declarations — `enum`,
`union`, `bitmask`, and `bitset` are encoded but do not yet get a generated constant.
`typedef`/alias TypeObjects remain deferred (encoder emits TK_NONE placeholder). See
[`features.md`](features.md).

## `--generate-interfaces`: DCPS API Generation

When `--generate-interfaces` is set, each IDL `interface` declaration and each
topic-type struct get typed DataWriter/DataReader wrappers in the
**language-idiomatic** form for the target backend.

**How topic types are determined:**
- A struct with `@topic` annotation: `TypeAnnotations.is_topic == true`.
- A struct without `@nested`: `TypeAnnotations.is_nested == false` (default).
- Structs with `@nested` are helper types — they do not get DataWriter/DataReader.

The full normative DCPS IDL is in [`dcps_idl.md`](dcps_idl.md).

**Per-language interface shapes:**

*Zig* — fat-pointer vtable structs (current):
```zig
// FooDataWriter — typed write operations
pub const FooDataWriter = struct {
    ptr: *anyopaque,
    vtable: *const Vtable,
    pub const Vtable = struct {
        register_instance: *const fn (*anyopaque, *const Foo) i32,
        write: *const fn (*anyopaque, *const Foo, i32) i32,
        dispose: *const fn (*anyopaque, *const Foo, i32) i32,
        // ...
    };
};
```

*C* — opaque typedef + free function declarations (planned; see roadmap):
```c
typedef struct FooDataWriter_s *FooDataWriter;
int32_t FooDataWriter_write(FooDataWriter writer, const Foo *data, int32_t handle);
```

DDS object entities are opaque handles; operations are free C functions.  This
matches the OMG C PSM binding and the idioms of major C DDS implementations.
Listener interfaces become plain C callback structs (a context pointer plus one
function pointer per callback), matching the same convention.

*C++11* — RAII wrappers using `std::string` / `std::vector` (planned).

*Java* — interfaces with JNI bridge (planned).

## `--generate-c-api`: C-ABI Bridge Layer (Zig backend, planned)

A companion flag to `--generate-interfaces` for the Zig backend.  Where
`--generate-interfaces` generates the idiomatic Zig vtable representation,
`--generate-c-api` additionally generates the `pub export fn callconv(.c)`
wrappers that implement the free functions declared by the C backend's
`--generate-interfaces` output.

For **DDS object interfaces** (runtime-implemented): generates a wrapper
function per operation that converts C-ABI types (`[*:0]const u8`, C structs)
to Zig-idiomatic types (`[]const u8`, Zig structs) and dispatches through the
Zig vtable.

For **listener interfaces** (user-implemented callbacks): generates a
`CXxxListenerAdapter` struct that holds the user's C callback struct and
implements the Zig listener vtable by converting DDS entity fat-pointers back
to C opaque handles before calling the C callbacks.  Adapter creation is
embedded in the DDS object C-ABI implementations (e.g. `create_datareader`
allocates the adapter when a non-null listener is supplied; the impl's `deinit`
frees it).

Listener interfaces are identified by name suffix (`*Listener`).

**Performance note:** The hot-path typed operations (`write`, `read`, `take`)
are generated per-type from the user's own IDL and use only C-compatible types
(pointer to data type, `InstanceHandle_t` integer).  There is zero conversion
overhead on the data path.  Conversion overhead applies only to cold-path setup
operations (topic name strings, filter expression strings, `StringSeq`
parameters), which are called at most once per entity during startup.

## DDS-RPC (Design Constraint)

DDS-RPC v1.0 (formal/17-04-01) enables request-reply patterns over DDS topics.
`--generate-interfaces` and DDS-RPC are intentionally orthogonal:

- `--generate-interfaces` emits the per-language idiomatic API shape.
- DDS-RPC request/reply pairs are a separate code-gen path (`--rpc`, future).
- IDL `interface` operations map to request/reply topic pairs in DDS-RPC, not to
  per-language dispatch — the two must not be conflated.

A future `--rpc` flag could emit request/reply struct types and serialization
instead.

## Non-CDR Encodings

JSON, XML, TOML, and other non-CDR encodings are not a code-generation concern. The
right approach is a schema-driven runtime that uses TypeObject as the schema descriptor:

- Every generated type carries pre-computed `type_object` bytes (Zig backend).
- A DynamicType/DynamicData implementation (owned by the forthcoming DDS runtime)
  parses those bytes and drives encoding-agnostic serialization.
- XCDR↔JSON, XCDR↔XML, MQTT bridge, config-file serialization etc. are all DDS runtime
  features, not zidl backend features.

DynamicType/DynamicData and the TypeObjectFactory belong in the DDS runtime, not in
zidl or `zidl-xtypes`. The `zidl-xtypes` package provides only the constants
(`EK_*`/`TK_*`/`TI_*`/flag values) shared by zidl and the DDS runtime.

## DDS Compliance Profiles

The DDS v1.4 spec defines five compliance profiles:

| Profile | Optional features added |
|---|---|
| Minimum | Mandatory DCPS only |
| Content-subscription | ContentFilteredTopic, QueryCondition, MultiTopic |
| Persistence | DURABILITY_SERVICE QoS, TRANSIENT + PERSISTENT durability kinds |
| Ownership | EXCLUSIVE_OWNERSHIP, OWNERSHIP_STRENGTH, HISTORY depth > 1 |
| Group access | PRESENTATION access_scope = GROUP |

zidl's `--generate-interfaces` targets the full DCPS API (all five profiles).
